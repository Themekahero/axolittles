## ⭐ Recommended order (highest impact first)

1. Make the Learn/Rhymes/Parents stage responsive (fluid layout or at minimum scale-to-width on portrait + a 'rotate to landscape' overlay matching the one Axo-Ninja already ships) so the core product is usable on a phone — the #1 thing blocking toddlers today.
2. Remove the external social links (Instagram/X/YouTube) and the 'Axo Rhymes on YouTube' link from the kid-facing Footer, and remove Shop/Parents from the in-game GameFrame nav — seal the walled garden so a tap never lands a 2-5yo on open social media or a catalogue.
3. Ship pre-recorded audio for the core spoken lines (letters, numbers 1-10, words, quiz prompts, celebration) with TTS as fallback, and add a one-tap welcome gesture that resumes AudioContext/warms speechSynthesis so the first lesson is never silent.
4. Fix the parental gate: replace the plaintext sum + 1.5s press-and-hold with a no-hint multi-digit numeric challenge that a toddler cannot pass by accident.
5. Either implement or REMOVE the three fake controls (playLimitMin screen-time timer, requirePurchaseApproval link/buy gate, musicOn background music) — never ship a parental control that no code reads.
6. Enable gzip/brotli for application/javascript and text/css on the live server (one-line config) so the 569KB JS isn't shipped raw — the single biggest, cheapest perf win on the audience's slow tablets.
7. Add a top-level React error boundary around RoutedShell and wrap all localStorage.setItem calls in try/catch so private-mode/quota errors degrade to a friendly 'tap to restart' instead of white-screening mid-lesson.
8. Set the correct production domain (VITE_SITE_URL) and regenerate canonical/OG/sitemap/robots; drop the price-0 'coming soon' Product/ItemList JSON-LD and the /category/* sitemap entries so search engines aren't told this is an active store.
9. Delete the dead storefront/blog/admin tree (AdminGate, BlogAdmin, StorefrontPages, src/auth, src/blog) and the axofamserver.onrender.com endpoint + admin login form from the bundle; remove the dead /cart, /wishlist, /track-order nav links that 404; remove the unused 2.48MB ComingSoon.png import.
10. Add sandbox='allow-scripts allow-same-origin allow-fullscreen' (deliberately NO allow-popups / allow-top-navigation) to the GameFrame iframe so embedded games physically cannot open external tabs, neutralizing the dormant window.open path before SOCIAL_LINKS is ever populated.

## 🔴 Blockers — fix before a toddler touches it (6)

- [ ] **B1. Core Learn experience letterboxes to a tiny ~390x244px strip on a phone in portrait — the most common kids device** &nbsp;`UX`
    - **Evidence:** src/learn/LearnApp.jsx:76-91 useStageScale = Math.min(innerWidth/1280, innerHeight/800); learn.css:53-57 .stage is a fixed width:1280px;height:800px scaled by transform. On 390x844 (iPhone 12/13/14) scale=0.305 → stage renders 390x244px with ~300px empty bands top+bottom. On 360x780 Android: scale=0.281 → 360x225px. All of /learn, lessons, FindIt quiz, /rhymes (VideoRoom), trophies render inside this box.
    - **Why it matters:** Toddlers and preschoolers are overwhelmingly handed a phone (and usually held in portrait by a parent). The entire learning product — its whole reason to exist — shrinks to a small centered card surrounded by useless empty space. A 2-5yo cannot reasonably parse or aim at a UI that is one-third the size it was designed at. This single issue makes the headline feature effectively unusable on the device most kids will use.
    - **Fix:** Make the stage responsive instead of a fixed scaled canvas. Either (a) re-author the Learn screens with fluid layout (flexbox/grid + clamp()/vw units, the way KidsHome.css and public/games/axo-arcade already do), or (b) as a minimum stopgap, switch useStageScale to scale to WIDTH on portrait phones (scale = innerWidth/1280) and let the stage scroll vertically, and/or add a portrait 'rotate to landscape' prompt like the Axo-Ninja game already has (public/games/axo-ninja/index.html:18-25).

- [ ] **B2. Tap targets fall far below toddler-safe minimums once the stage is scaled on phones** &nbsp;`UX`
    - **Evidence:** learn.css:119 .btn-round is 92px; .bottom-nav nav-pill is height:54px (learn.css:137); lesson word 58px (learn.css:341). At phone portrait scale 0.305 these become 28px buttons, 16px-tall nav pills, 18px text; at 0.281 (small Android) 26px / 15px / 16px. Apple HIG minimum is 44px, Google Material 48px.
    - **Why it matters:** 2-5yos have the poorest fine-motor precision of any user group; tap targets for them should be LARGER than the 44-48px adult minimum, not 26-28px. The bottom nav (Home/Learn/Games/Adventure/Rhymes/Shop/Settings) at ~15-16px tall is essentially un-tappable for a toddler and will cause constant mis-taps (e.g. hitting Shop or Settings instead of Home). This is a direct consequence of finding 1 but is independently disqualifying for the audience.
    - **Fix:** Resolve via responsive layout so controls keep their physical size. If keeping the scaled stage short-term, scale to width on phones so the rendered button size roughly doubles, and audit every interactive element to ensure >=48px rendered at the smallest supported viewport.

- [ ] **B3. Footer ships always-visible social links (Instagram/X/YouTube) that escape the walled garden in one tap** &nbsp;`SAFETY`
    - **Evidence:** src/Components/Footer/Footer.jsx:4-20 hardcodes socialLinks = [Instagram https://www.instagram.com, YouTube https://www.youtube.com/channel/UC4H2..., X https://x.com/Axolittles], rendered at lines 79-101 as <a target="_blank" rel="noreferrer">. The Footer renders on every non-immersive route (App.jsx:187). Confirmed shipped in the live build: grep of dist/assets/*.js returns instagram.com (1), x.com (2), youtube.com/channel (3).
    - **Why it matters:** For a 2-5yo, a brightly-colored icon at the bottom of the page is just another button. One tap opens Instagram/X/YouTube — uncurated adult social platforms with infinite scroll, comments, DMs, ads, and no content controls. This is the single most dangerous pattern on a toddler site: it defeats the entire 'safe walled garden' premise the platform is sold on. It also makes the COPPA 'child-directed' classification indefensible because you are funneling kids to third-party data collectors.
    - **Fix:** Remove the social icons from the kid-facing footer entirely, OR move them behind the parents gate / a 'grown-ups' zone only. If brand social must appear, render it only on a parent-facing page reached after the parental gate, never on routes a child uses. Never use target=_blank to external social on a child route.

- [ ] **B4. Footer 'Axo Rhymes on YouTube' link sends children directly to youtube.com** &nbsp;`SAFETY`
    - **Evidence:** src/storefront/navigationData.js:283 defines { label: "Axo Rhymes on YouTube", href: "https://www.youtube.com/channel/UC4H2mCujSLwz3H9It75-S4A", external: true }. Footer.jsx:104-126 renders footerColumns links with target={link.external ? "_blank"} — so this opens the open YouTube site in a new tab.
    - **Why it matters:** Same walled-garden breach as the social icons but worse because it is labelled with the child-recognizable brand 'Axo Rhymes', so a child who knows the rhymes will deliberately tap it — landing on full youtube.com with recommendations, autoplay to arbitrary channels, comments and ads. The whole point of the in-app /rhymes room (nocookie, curated) is undermined by a one-tap door to the raw platform.
    - **Fix:** Delete this external link from navigationData footer columns, or relabel and relocate it behind the parents gate. Keep children inside the in-app /rhymes VideoRoom.

- [ ] **B5. Privacy Policy is not COPPA/GDPR-K compliant and misdescribes the actual service** &nbsp;`SAFETY`
    - **Evidence:** src/Components/LegalPages/PrivacyPolicyPage.jsx: 'Information We Collect' claims IP address, device info, usage/clicks (lines 9-15); 'Cookies and Analytics' claims cookies + analytics tools (lines 56-65); the 'Children's Privacy' section (lines 89-105) is generic boilerplate: 'We encourage parents to supervise... We do not knowingly collect personal information from children except where permitted by applicable law.' There is NO COPPA direct-notice, NO description of verifiable parental consent (VPC), NO age screen, and NO list of the third parties (YouTube, Cloudinary, ImageKit, Google Fonts, jsDelivr/cdnjs) that receive each child's IP. Meanwhile the app itself wires no analytics (grep for gtag/GA/pixel/segment/mixpanel/etc. in src = none) and the contact form is a no-op (ContactPage.jsx:29-32 just setStatus('sent')).
    - **Why it matters:** A service this clearly directed to ages 2-5 is squarely under COPPA (US) and the GDPR-K / UK Age-Appropriate Design Code. COPPA requires, before any collection from under-13s, a clear direct notice of exactly what is collected and a VPC mechanism — neither exists. The policy simultaneously over-claims (says it uses analytics/cookies it doesn't actually run, which is a false disclosure) and under-discloses (omits that embedding YouTube/Cloudinary/ImageKit transmits the child's IP and lets third parties set cookies). Either gap alone is an FTC/ICO enforcement risk; together they make the policy unfit to publish on a kids' site.
    - **Fix:** Engage privacy counsel and rewrite as a child-directed policy: explicitly state the app stores only on-device localStorage and collects no account/PII; enumerate every third-party embed (YouTube-nocookie, Cloudinary, ImageKit, Google Fonts, Font Awesome/jsDelivr) and what each receives (IP, request data); add a parental-rights/contact section and, if any data is ever collected, a real VPC flow and age gate. Remove the false 'we use analytics/cookies' language unless/until you actually deploy a child-appropriate, consent-gated analytics setup.

- [ ] **B6. Live server serves JS and CSS uncompressed (no gzip/brotli on static assets)** &nbsp;`PERF`
    - **Evidence:** curl -sI -H 'Accept-Encoding: gzip, br' https://axolittles.cterminal.xyz/assets/index-DT_kLLZg.js returns Content-Length: 568763 with NO content-encoding header; same for the CSS (Content-Length: 212691, no encoding). The root HTML IS gzipped (content-encoding: gzip), proving the client sends Accept-Encoding and the server can compress — but it omits application/javascript and text/css from gzip_types. So the real transfer is 569KB JS + 212KB CSS = ~780KB raw, not the build's reported 184KB+40KB gzip.
    - **Why it matters:** Kids overwhelmingly use tablets/phones, often on home wifi shared with a streaming TV or on a parent's phone hotspot. ~780KB of uncompressed blocking resources before first paint is roughly 4x the transfer it should be. On a mid-range Android tablet over a weak connection this adds many seconds to a blank screen — toddlers and the parents wrangling them will bounce. This is a one-line server config fix that quadruples effective load speed, so shipping without it is leaving the single biggest win on the table.
    - **Fix:** Enable gzip (and ideally brotli) for application/javascript, text/css, image/svg+xml, application/json in the nginx/server config (gzip_types ...; gzip on; or brotli on). Verify with curl that content-encoding: gzip/br appears on the .js and .css responses. No code change needed; redeploy not required, just server config + reload.

## 🟠 Major — fix before public launch (28)

- [ ] **M1. No portrait/orientation prompt or fullscreen for the React Learn stage (the embedded game has one, the core app doesn't)** &nbsp;`UX`
    - **Evidence:** grep for orientation/portrait/rotate across src/ returns ONLY LearnApp.jsx:84,87 (a resize listener that recomputes scale). public/games/axo-ninja/index.html:18-25 ships a #rotate-overlay 'Rotate your device to landscape to play' and main.js:608 tryLockLandscape(); the React /learn stage has neither. No requestFullscreen exists for the React stage (only iframes in GameFrame.jsx:51 and VideoRoom.jsx:85).
    - **Why it matters:** The product is inconsistent: tap Adventure and a kid is told to rotate; tap Learn and they silently get a tiny box with no guidance. A parent has no signal that landscape (where scale jumps to 0.49 and buttons reach ~45px) is dramatically better. Kids and parents will conclude the app is broken.
    - **Fix:** Add the same rotate-to-landscape overlay used by Axo-Ninja to the immersive .axo-learn--immersive routes, and/or trigger requestFullscreen on first interaction so the stage uses the full screen. Better: fix responsiveness (finding 1) so orientation matters less.

- [ ] **M2. All spoken guidance depends on Web Speech TTS with no audio fallback — silent for many kids** &nbsp;`UX`
    - **Evidence:** src/learn/audio.js:54-65 speak() returns silently if !('speechSynthesis' in window) or if englishVoices is empty; it relies on system voices loaded async via onvoiceschanged (audio.js:42-43). LessonScreen.jsx:31 calls AxoAudio.speak on mount with no user gesture; the only bundled audio asset is public/bg-song.mp3 — there are NO pre-recorded letter/word/number clips. Browsers commonly block speechSynthesis until a user gesture, and many Android/iOS WebView/in-app-browser contexts have no usable en voice.
    - **Why it matters:** For a NON-READER, the voice IS the lesson — 'A for Apple', counting '1, 2, 3', 'Find the cat'. If TTS is silent (very common in iOS in-app browsers, some Android Chrome configs, or before first tap), the child sees a letter and a picture with no instruction and no feedback, and the FindIt quiz prompt ('Find the ___') is spoken-only — the on-screen text is unreadable to the target age. The experience degrades from 'learning app' to 'mute picture box' with no warning.
    - **Fix:** Ship pre-recorded audio for the core spoken lines (letters, words, numbers 1-10, quiz prompts, celebration) and prefer those over TTS, falling back to TTS only when missing. At minimum, detect 'no voice available / speech blocked' and surface a one-time parent-facing notice, and ensure the FindIt quiz never depends on audio alone (the target word/letter is already shown — make the visual prompt prominent and add a persistent 'tap to hear' that works after the first gesture).

- [ ] **M3. Bottom navigation exposes Shop and Settings/Parents directly to the child with toddler-sized mis-tap risk** &nbsp;`UX`
    - **Evidence:** ui.jsx:587-595 BOTTOM_NAV_ITEMS includes Shop (onShop -> /shop) and Settings (onSettings -> /parents) shown on Home/Learn/Trophies/Rhymes; LearnApp.jsx:194-195 wires them to route straight out. HomeHub.jsx:99 also surfaces a Shop feature card. The /parents route does have a math/hold gate (ParentsDashboard.jsx:87-180), but Shop has no gate and the purchase-approval toggle (ParentsDashboard.jsx:430-438, default requirePurchaseApproval:true) is described in-code as display-only.
    - **Why it matters:** A 3-year-old randomly tapping the tiny nav bar will frequently land in the Shop (a full AxoFam catalogue in a different visual style) or bounce against the Parents gate. Jumping a non-reader into a product catalogue is confusing and, for a kids product, a soft commerce-safety concern; the catalogue is currently price-0/'coming soon' so no real purchase risk today, but the pattern is wrong for the audience.
    - **Fix:** Remove Shop and Settings from the CHILD-facing bottom nav (keep them behind the Parents area or a grown-up corner). If Shop must stay reachable, route it through the same grown-up gate, and make the requirePurchaseApproval gate actually intercept buy/checkout rather than being display-only before any real commerce ships.

- [ ] **M4. Embedding third-party YouTube in Axo Rhymes still leaks data and exposes escape paths despite nocookie+rel=0** &nbsp;`SAFETY`
    - **Evidence:** src/learn/screens/VideoRoom.jsx:11-12 embeds https://www.youtube-nocookie.com/embed with params rel=0&modestbranding=1&iv_load_policy=3 in an <iframe allow="autoplay; encrypted-media; fullscreen"> (lines 81-87). The 'Play all' button (lines 39-48) streams the channel uploads playlist (data.js:417-421). Thumbnails are loaded from https://i.ytimg.com (line 66), which contacts Google before any video is played. Note: modestbranding is deprecated/ignored by YouTube, rel=0 only restricts related videos to the same channel — it does not remove ads, the YouTube logo, the in-player 'Watch on YouTube' / title link to youtube.com/watch, or end-screen/info cards.
    - **Why it matters:** youtube-nocookie reduces but does not eliminate tracking: ytimg/Google still receives the child's IP and player-side cookies/local storage can still be set on first play. The player title and logo remain tappable doorways to youtube.com, and YouTube serves ads in embeds it does not classify as 'made for kids' — meaning a 2-5yo can both see ads and tap out to the open platform mid-song. Relying on a third-party player you do not control for a COPPA child-directed surface is the exact pattern the FTC penalized in the 2019 YouTube/Google settlement.
    - **Fix:** Strongest: self-host the Axo Rhymes videos (Cloudinary/your own CDN, served same-origin) and play them in a native HTML5 <video> with no external player chrome. If YouTube must stay short-term: keep youtube-nocookie, but disclose it in the policy, add a visible 'this opens videos from YouTube' note on the parents side, and accept it as an interim measure flagged for replacement before public launch. Do not pre-load i.ytimg thumbnails for child users without disclosure.

- [ ] **M5. Parents gate is trivially bypassable by a toddler and gates controls that aren't even enforced** &nbsp;`SAFETY`
    - **Evidence:** src/learn/screens/ParentsDashboard.jsx: the gate shows the literal sum in plaintext — 'Solve to continue: {challenge.a} + {challenge.b} = ?' (line 133) — and offers a press-and-hold fallback that requires ZERO knowledge: holding the button for HOLD_MS=1500ms (lines 39, 105-115) calls onUnlock(). Separately, the controls behind the gate are decorative: requirePurchaseApproval and playLimitMin are defined in DEFAULT_SETTINGS (lines 30-36) and mirrored in LearnApp.jsx:32-33, but grep shows they are never read/enforced anywhere outside the dashboard that sets them. The screen-time section even admits it: 'this reminder is display-only for now — it does not lock the app or stop play' (line 407).
    - **Why it matters:** A 1.5-second press-and-hold is something a 2-year-old does by accident; it is not a parental gate by any reasonable standard (Apple/Google kids guidelines require a gate a young child cannot pass, e.g. a typed multi-digit answer with no on-screen hint, not a hold). So 'Grown-ups only' is a false assurance. Worse, the 'Require grown-up approval before opening shop links / checkout' toggle (lines 431-437) tells parents purchases are protected when nothing in code enforces it — a trust-eroding misrepresentation, and a real risk the moment buy links are turned on.
    - **Fix:** Replace the gate with a genuine adult check: a randomized multi-digit numeric entry with NO on-screen hint (e.g. 'type 4 7 2'), or a 'type the number written in words' challenge; remove the zero-knowledge press-and-hold bypass. Then actually wire the settings: have any future buy/checkout link and the play-timer read axolittles-settings and enforce requirePurchaseApproval/playLimitMin, or remove those toggles until they do something. Do not show controls that imply protection that isn't implemented.

Remove the dead axofamserver.onrender.com reference from the shipped bundle.

## ⚖️ Calibration — items the audit OVERSTATED (deprioritize, don't drop)
_Kept for honesty; treat these as lower priority than their original severity implied._

- **Privacy Policy 'over-claims analytics/cookies it does not run' framed as a standalone false-disclosure violation** — The underlying facts are right (grep confirms zero analytics wired in src) and disclosing the third-party CDN IP exposure IS a real gap. But the 'false disclosure' angle is overstated: privacy policies routinely describe categories of data the operator MAY collect, and over-disclosing capabilities you don't currently use is far less of an enforcement risk than under-disclosing. Keep the real finding (no COPPA direct notice, no VPC, no age gate, undisclosed third parties) and drop the 'they lie about using analytics' characterization, which weakens credibility.

## 🎯 Biggest risks (one-line summary)

- The headline product is unusable on the audience's actual device. The entire React Learn/Rhymes/Trophies/Parents stage is a fixed 1280x800 canvas scaled by min(w/1280, h/800) (LearnApp.jsx:80) — on a portrait phone it collapses to a ~390x244 letterboxed island with toddler tap targets shrinking to ~26-28px. There is no portrait prompt and no fullscreen for the React stage (unlike the embedded Axo-Ninja game, which DOES ship a rotate overlay). A parent handing a 3yo a phone gets a tiny, mute, confusing box. This single issue makes the core feature effectively broken on the most common kids device.
- Parental controls are theater AND the gate is trivially bypassable — a double trust failure on a kids product. Verified: the gate prints the literal sum '6 + 7 = ?' (ParentsDashboard.jsx:133) and offers a 1.5s press-and-hold (HOLD_MS=1500, line 39/110) that any toddler triggers by accident. Worse, grep confirms playLimitMin, requirePurchaseApproval, and musicOn are NEVER read by any consumer (only declared in two DEFAULT_SETTINGS blocks + the dashboard toggles). Parents who set a screen-time limit and 'require approval before purchases' get false safety assurance that does literally nothing.
- The walled garden is not sealed. The shared Footer (rendered on /, /shop, /about, etc.) hardcodes target=_blank links to instagram.com, x.com/Axolittles, youtube.com/channel and a kid-recognizable 'Axo Rhymes on YouTube' link — one tap drops a 2-5yo onto open adult social media. The /rhymes player uses youtube-nocookie+rel=0 but still loads i.ytimg thumbnails (Google gets the child's IP before play) and keeps the tappable YouTube logo/title. This defeats the entire 'safe walled garden' premise and makes the COPPA child-directed classification indefensible.
- Audio — the entire teaching mechanism for non-readers — has zero fallback. AxoAudio.speak (audio.js:54-65) silently returns if speechSynthesis is missing or English voices haven't loaded; LessonScreen auto-speaks on mount before any user gesture; the FindIt quiz prompt ('Find the cat') is spoken-only. On the common iOS in-app-browser / locked-tablet / pre-gesture cases the lesson is a silent, instruction-less picture box. No pre-recorded clips exist as backup (only bg-song.mp3).
- No error boundary anywhere + unguarded localStorage writes = total white-screen crash. Confirmed no ErrorBoundary/componentDidCatch in src; every star earned triggers an unguarded localStorage.setItem(PROGRESS_KEY) (LearnApp.jsx:137). In Safari Private Browsing or at quota, setItem throws QuotaExceededError and unmounts the whole app to a blank page with no recovery — mid-lesson, for a toddler.
- Production hygiene is broken in launch-sabotaging ways: live canonical/OG/sitemap/robots all point to the WRONG domain axolittles.com (verified on the live cterminal.xyz site), and the live server ships JS uncompressed (verified: /assets/index-DT_kLLZg.js returns 568763 bytes with NO content-encoding despite the HTML being gzipped) — quadrupling effective load on the cheap tablets/cellular this audience uses.
